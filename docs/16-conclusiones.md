# 16. Conclusiones

## 16.1 Resultados y análisis del proyecto

### 16.1.1 Logros obtenidos

- Se definió una estructura técnica adaptada a la realidad de Conmaquel, cubriendo sus necesidades actuales.
- Se diseñó un sitio web y sistema de gestión que integra atención al cliente, ventas, inventario y servicios.
- Se establecieron roles claros para evitar errores en el manejo de la información.
- El uso de Scrum permitió ir validando avances con el cliente en cada etapa, asegurando que el producto final sea útil.
- Se construyó un prototipo navegable completo (25 pantallas) que cubre el módulo público, autenticación y los nueve módulos del panel administrativo, con un sistema de diseño consistente listo para su implementación en HTML, CSS, JavaScript, PHP y MySQL.

### 16.1.2 Aspectos a mejorar

- Definir una duración fija de 2 semanas por Sprint para mayor uniformidad.
- Agregar criterios de aceptación más detallados para cada historia de usuario (ver ejemplo aplicado en **05-historias-de-usuario.md**).
- Incluir en el plan las reuniones formales de revisión y retrospectiva de cada ciclo.

### 16.1.3 Influencia del modelo Scrum

- **Flexibilidad**: permitió ajustar requisitos cuando se agregaron funcionalidades para el control de servicios técnicos.
- **Transparencia**: las reuniones diarias mantuvieron informados tanto al equipo como al cliente.
- **Entrega de valor**: al final de cada Sprint se cuenta con una parte funcional, lista para probar o usar.

## 16.2 Conclusiones

La adaptación del proyecto a Conmaquel bajo el modelo Scrum ha resultado muy adecuada. Al contar con un sitio web y sistema de gestión propio, la empresa dejará de usar métodos manuales y podrá operar con mayor orden, rapidez y control. El enfoque ágil permitirá seguir agregando mejoras en el futuro sin tener que rehacer todo el trabajo. El diagnóstico organizacional y la evaluación comparativa de modelos confirmaron que Scrum era la opción más consistente con la cultura, la madurez tecnológica y las restricciones de negocio de la empresa (ver **03-metodologia-scrum.md**).

## 16.3 Limitaciones del plan y propuestas de mejora

- Al ser un equipo de un solo integrante, la separación de roles Scrum (Product Owner, Scrum Master, equipo de desarrollo) es más conceptual que real, lo que puede generar sesgos en la priorización del backlog. Se propone incorporar, en una siguiente etapa, revisiones cruzadas con un tercero (docente o par académico) que cumpla el rol de validador externo.
- El sistema de métricas propuesto (velocity, burndown, cumplimiento de sprint) aún no cuenta con datos históricos de referencia, por lo que los umbrales de alerta definidos deberán calibrarse con la información real de los primeros sprints ejecutados.
- El plan de calidad no incluye todavía pruebas de carga ni de penetración formales, dado el alcance académico del proyecto; se recomienda incorporarlas antes de un eventual paso a producción real con datos de clientes.
- La estimación de duración de cada sprint se basó en la complejidad relativa del backlog y no en velocity histórica del equipo, por lo que los tiempos podrían ajustarse una vez ejecutado el Sprint 0.

## 16.4 Recomendaciones

- Mantener ciclos cortos de trabajo para responder rápido a cambios en el mercado.
- Capacitar al personal antes de la puesta en marcha para que usen el sistema con confianza.
- Registrar todas las modificaciones para facilitar actualizaciones futuras.
- Considerar agregar la integración con facturación electrónica SUNAT en versiones posteriores.
- Evaluar la incorporación de una pasarela de pago en línea para el módulo de ventas y reservas, una vez estabilizado el sistema base.
